# Speaker Script: No-Nonsense Agentic AI

## Part 1: Introduction & The Architectural Spectrum (8 Mins)

### Slide 1: Welcome & The Frame

**Speaker:** "Hey everyone, thanks for coming. Today, we’re doing a strictly 'no-nonsense' dive into Agentic AI. If you've looked at this space recently, it feels like it's dominated by heavy, complex software frameworks. You install a library, write three lines of magic code, and suddenly an 'agent' is running. But as engineers, we hate magic boxes because when they break in production, we don't know how to fix them. Today, we are stripping away all the wrappers. We are going to build an agent using nothing but raw code and standard system architecture."

### Slide 2: GenAI vs. AI Agents vs. Agentic AI

**Speaker:** "Before we write code, let’s clear up the buzzwords. The industry has moved across a spectrum, and we need to understand the structural differences.

First, we have **Generative AI**. This is completely passive and stateless. Think of standard ChatGPT. You send a prompt, the LLM processes it, sends a response back, and the transaction is dead. It’s just a text transformer.

Next up are basic **AI Agents**. Here, we wrap that text transformer in a bit of code so it can achieve a specific task. We give it a couple of database tools. For example: a user asks to check an order status, the code intercepts the request, lets the LLM choose a database tool, fetches the ID, and returns the result. It’s useful, but the path is tightly scoped.

Finally, we have **Agentic AI**. This is where things get interesting. We aren't just giving the LLM a tool; we are giving it *autonomy*. Agentic systems manage their own non-linear workflows. If a tool returns an error, the system analyzes the error and tries a different approach. It manages its own long-term memory, decides when to pause for a human, and runs in an open-ended loop until a complex goal is achieved."

### Slide 3: The Four Primitives

**Speaker:** "If you look under the hood of *any* agentic system—no matter how fancy the marketing is—it is always made of the exact same four internal components:

1. **The Brain:** The core LLM. Remember, the LLM itself is frozen weights. It doesn’t 'do' anything until we call its API. It’s simply an on-demand pattern matcher.
2. **The Memory Engine:** The data structures we build to hold session history and background facts.
3. **The Planning Loop:** The system prompts and logic patterns that force the LLM to think before it acts.
4. **The Tools:** Standard code functions—like database drivers or HTTP clients—that our code executes on behalf of the LLM."

## Part 2: The Core Execution Engine & The ReAct Loop (10 Mins)

### Slide 4: The ReAct Pattern

**Speaker:** "How do we make a frozen LLM behave like an autonomous actor? We use an execution cycle called the **ReAct pattern**, which stands for **Reasoning and Acting**. It breaks down into four alternating phases:

* **Perceive:** Our code takes the user's goal plus whatever happened in the last turn and feeds it into the LLM's context window.
* **Think:** We explicitly instruct the LLM to output its 'thoughts' in natural language first. It has to explain *why* it wants to take an action before taking it.
* **Act:** The LLM outputs a machine-readable command, usually a structured JSON block specifying a tool name and arguments.
* **Observe:** Our code parses that JSON, runs the actual tool in our environment, gets the raw output, and passes it right back to the *Perceive* step. Then the loop repeats."

## Part 3: Standardized Tool Integration & The MCP Layer (10 Mins)

### Slide 6: The $N \times M$ Bottleneck

**Speaker:** "Once you have a loop running, you want your agent to interact with your company's infrastructure. Historically, this is where engineering teams hit a massive wall. If you have $N$ different LLM models you want to test, and $M$ different internal microservices, databases, and APIs, you end up writing custom, bespoke integration code for every single combination. It's incredibly fragile, and if an API schema changes, the agent crashes."

### Slide 7: Enter Model Context Protocol (MCP)

**Speaker:** "To solve this, we use an open networking standard called **Model Context Protocol**, or MCP. MCP is not a framework; it's a lightweight, stateful JSON-RPC 2.0 protocol. Think of it like a universal USB-C port for AI models.

The architecture splits into three pieces:

* **The Host:** This is the core application loop we just built.
* **The Client:** A standardized connection block inside our host that knows how to talk to MCP servers.
* **The Server:** A lightweight, isolated microservice that wraps your database or internal API. The server doesn't know or care *which* LLM is running; it simply exposes functions using standard primitives: `tools` for executable actions, `resources` for raw data like logs, and `prompts` for pre-made templates." ## Part 4: The Memory Matrix & Production Hardening (10 Mins)

### Slide 9: Short-Term vs. Long-Term Memory

**Speaker:** "Now that our agent can run in a loop and talk to databases, it faces a memory problem. If an agent runs for 20 turns, the chat history gets massive. This brings us to the **Memory Matrix**.

First, we have **Short-Term Context Memory**. This is the active, in-flight transaction history of our current loop. To prevent it from overflowing the model's token limits and costing a fortune, we implement Sliding Window Trimming/Deletion/**Recursive Summarization**. Our coordinator code monitors the token count. The moment it hits a threshold, we pass the oldest logs to a background LLM call, compress them into a brief 'Current Status Summary,' swap it into the prompt, and discard the raw, chatty middle turns.

Second, we have **Long-Term Fact Memory**. If the agent learns something permanent—like 'User prefers data sorted by timestamp'—we don't keep that in the chat log. Our code saves that fact to a structured JSON file or key-value profile. Every time this agent boots up, that profile is automatically appended to its system instructions.

Third, we have **RAG (Retrieval-Augmented Generation)**. We treat RAG strictly as an external lookup tool via MCP. If the agent calculates that it lacks domain knowledge during its *Think* phase, it explicitly calls a vector search tool to fetch exact text blocks."

### Slide 10: Long-Term State Management & HITL

**Speaker:** "The final piece of production hardening is dealing with infrastructure realities. Runtimes crash, servers reboot, and autonomous agents shouldn't be executing high-stakes code blindly.

We use **Long-Term State Management (LTSM)**, or State Checkpointing. At the end of every single loop iteration, our code writes the entire agent state—its context thread, loop counter, and metadata—to a persistent SQL database row. If the container dies, we don't lose the agent's progress.

This database persistence is what makes **Human-in-the-Loop (HITL)** possible. When the agent wants to perform a destructive action, like running a database delete or authorizing a payment, our control loop catches it, pauses the thread, saves the state to the DB via LTSM, and goes to sleep. It exposes that task to an external dashboard queue and waits for an asynchronous human signal to approve, reject, or modify the payload before waking up and resuming."

### Slide 11: Live Demo 3 — Resiliency & Approvals

**Speaker:** *"Let’s see it in action. I'm going to ask the agent to optimize a database index, which involves a write mutation. That is resilient, framework-free agent engineering."*

## Part 5: The Bottom Line & Q&A (7 Mins)

### Slide 12: The Reality Check

**Speaker:** "To close things out, let’s talk about the brutal realities of putting agents in production.
First, there is a **Latency Tax**. Because these loops are sequential and multi-turn, an agent taking 5 steps to solve a problem means 5 consecutive network roundtrips to an LLM provider. This takes seconds, not milliseconds. If your user experience requires instant responses, an autonomous agent is the wrong architecture.

Second, beware the **Agent Anti-Pattern**. Just because you *can* use an autonomous loop doesn't mean you *should*. If a business workflow is entirely predictable, do not use an unpredictable LLM loop to figure out the steps. Write a fast, 100% reliable Directed Acyclic Graph (DAG) or a standard deterministic script. Use agents only when the path to the solution is highly dynamic and unknown at compile time.

Strip away the framework hype, build your loops manually, standardize your interfaces with open protocols like MCP, and treat state management like classical software engineering. Thanks for your time, let's open the floor for questions."
